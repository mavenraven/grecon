# grecon

A tmux-native dashboard for managing [Claude Code](https://claude.ai/claude-code) sessions.

Run multiple Claude Code instances in tmux, then manage them all from one place — see what each agent is working on, which ones need your attention, switch between them, kill or spawn new ones, and resume past sessions.

```
┌ grecon — Claude Code Sessions ────────────────────────────────────────────────────────────────────────┐
│ Session              Directory           Status  Summary                                              │
│ api-refactor         ~/repos/myapp       ● Input Waiting for approval to delete legacy auth endpoints │
│ debug-pipeline       ~/repos/infra       ● Work  Fixing flaky CI by adding retry logic to deploy step │
│ write-tests          ~/repos/myapp       ● Work  Adding integration tests for the new auth middleware │
│ code-review          ~/repos/webapp      ● Idle  PR #452 ready for review, all checks passing         │
│ scratch              ~/repos/recon       ● Idle  Updated README with new summary feature              │
└───────────────────────────────────────────────────────────────────────────────────────────────────────┘
j/k navigate  Enter switch  x kill  / search  i next input  q quit
```

- **Input** rows are highlighted — these sessions are blocked waiting for your approval
- **Working** sessions are actively streaming or running tools
- **Idle** sessions are done and waiting for your next prompt

## AI summaries

Each session gets a one-line summary of what the agent is currently doing, generated by Haiku. The server extracts recent activity from each session's JSONL (text output + tool usage since the last user message), then calls `claude -p --model haiku` to summarize it.

Summaries update lazily — only when the JSONL content actually changes. If the Haiku call fails (network, rate limit, etc.), the old summary stays or shows a dash. No crashes, no retries.

This is only possible because grecon uses a persistent server. A standalone TUI would have to re-discover sessions and parse JSONL on every render tick — there's no room for async LLM calls. The server polls once every 2 seconds, fires off summary requests in the background, and serves cached results instantly to any client that connects.

## How it works

grecon runs a background server that polls tmux every 2 seconds. The TUI connects to this server over a Unix socket.

```
tmux list-panes (#{pane_pid})  →  PID → tmux session name
~/.claude/sessions/{PID}.json  →  PID → JSONL session ID
~/.claude/projects/*/*.jsonl   →  session ID → tokens, model, timestamps
tmux capture-pane              →  tmux session → status (last line of pane)
~/.claude/projects/*/*.jsonl   →  last assistant activity → Haiku → summary
```

Status detection reads the Claude Code status bar at the bottom of each tmux pane:

| Status bar text | State |
|---|---|
| `esc to interrupt` | **Working** |
| `Esc to cancel` | **Input** |
| anything else | **Idle** |

## tmux config

```bash
# prefix + g → switch to grecon (creates session if needed)
bind g run-shell 'tmux kill-session -t _grecon 2>/dev/null; tmux new-session -d -s _grecon grecon; tmux switch-client -t _grecon'

# prefix + f → jump to next session waiting for input
bind f run-shell 'grecon next'
```
